# 程序设计 写作指南

## 1. 核心任务

本章需系统呈现如何将教学需求转化为程序功能，重点展示"痛点→设计→功能"的映射关系。

## 2. 内容要素

### 2.1 需求分析

呈现教学需求调研的结果（通常包含表格）：
- 一线教师/学生的核心需求（访谈、问卷或教学经验归纳）
- 功能优先级分类（参考期刊模板.md中的表1：P0/P1/P2）

### 2.2 总体设计思路

概述程序的设计原则（2-3个）：
- 针对"演绎推理困难"：程序如何将抽象推理转化为可操作体验
- 针对"缺乏应用场景"：程序如何拓展教学案例的复杂性
- 双模式设计（验证 vs 探究）如何实现因材施教

### 2.3 功能模块设计

每个核心模块用一段话描述（可参考期刊模板.md中的表2）：

| 模块名称 | 核心功能 | 解决的教学痛点 |
|---------|---------|--------------|
| 系谱图绘制模块 | 交互式SVG系谱图绘制 | 传统板书不够直观 |
| 遗传规则引擎 | 5种遗传模式的计算 | 学生难以手动计算 |
| 多家族模块 | 复杂家系联姻 | 真实场景缺乏 |
| 电泳图模块 | DNA条带可视化 | 微观机制难理解 |

### 2.4 技术架构（可选）

如篇幅允许，可用一段话描述技术选型：
- 前端：React + SVG（交互性 + 可视化）
- AI 辅助：Google AI Studio / Gemini（代码生成）
- 优势：跨平台、无需安装

## 3. 结构规范

建议结构：
```
2 基于问题分析的程序设计
  2.1 需求分析
  2.2 总体设计思路
  2.3 功能模块设计
  2.4 技术架构（可选）
```

## 4. 常见问题

| 问题 | 解决 |
|------|------|
| 功能模块写成用户手册 | 以"解决什么教学问题"为切入点，而非功能清单 |
| 技术细节过多 | 面向一线教师，技术细节放在小结或注释中 |
| 与后面章节重复 | 本章聚焦"应该有什么功能"，第3章聚焦"怎么实现" |
